doc
La carpeta Doc incluye toda la documentación necesaria para trabajar con la plantilla. Se adapta a cada proyecto.
gulp
Es la carpeta donde están las tareas de gulp y sus archivos de configuración. Para ver más en profundidad cada archivo ver la sección gulp.
src
La carpeta src contiene todo el código con el que trabajamos. En su interior tenemos:
  • img
    • Contiene las imágenes del proyecto. Las cuales procesaramos antes de pasarlas a la carpeta final.
  • includes
    • Dentro van todos aquellos archivos html que luego incluimos dentro de un archivo general como index.html. Un nav, slider, sidebar, article... La idea es separar el html en archivos independientes que lo hacen más reutilizable.
  • js
    • Carpeta que contiene los archivos javascript. Dentro de la misma podemos crear una subcarpeta de nombre vendor donde ubicar los los archivos de terceros (jquery por ejemplo).
  • sass
    • Todo lo relacionado con los estilos. Para ver más en detalle cada subcarpeta ir al asección css.
.babelrc
Archivo necesario para trabajar con babel. Contiene un json con información sobre los presets que vamos a utilizar. Los presets són un conjunto de plugins.
.gitignore
Archivo responsable de comunicar a git que archivos o carpetas no debe incluir. En este caso le decimos que ignore la carpetas dist y node_modules. La primera es la carpeta que contiene los archivos ya procesados y la segunda contiene todos los paquetes descargados por npm. Ambás las generamos usando gulp y no necesitamos git en ellas.
.jshintrc
Archivo donde especificar las opciones para el linter jshint.
.stylelintrc
Archivo donde especificar las opciones para el linter stylelint.
gulpfile.babel.js
Gracias a babel podemos usar gulp con ecmascript6. Es el archivo propio de gulp (gulpfile) modificado para usar con babel.
humans.txt
Propuesta interesante que nace para dar importancia a las personas que trabajan en cada proyecto. No es obligatorio, pero es una manera de informar de la autoría. Sobretodo en aquellas situaciones donde trabajamos con intermediarios.
package.json
El archivo generado con gulp. Contiene información sobre las dependencias del proyecto. Así como el nombre, descripción e incluso su repositorio git.
README.md
Escrito en markdown es una información general de como instalar y usar la plantilla para mostrar en sitios como github.
robots.txt
Archivo opcional en cualquier proyecto. Se utiliza para controlar el rastreo de la web por parte de los motores de busqueda. Podemos encontrar mucha información al respecto en las herramientas de google

Gulp

Como instalar y usar gulp con la plantilla IniCio

Instalación

La plantilla trabaja con la versión 4 de gulp. Antes de instalar gulp4 hay que desisntalar cualquier versión previa ( Doy por hecho que node.js y npm están instalados ).


npm uninstall gulp -g

Con gulp 4 necesitamos instalar CLI que ahora esta separado del propio gulp.


npm install gulp-cli -g

Ahora gulp esta instalado globalmente. Descargamos la plantilla y ejecutamos:


npm install

El comando instalará todas las dependencias especificadas en package.json. Si todo es correcto ejecutando gulp -v veremos algo como esto:


CLI version 1.4.0
Local version 4.0.0-alpha.2 //solo en el caso de estar en el proyecto

Usando Gulp

Dentro de la carpeta gulp tenemos las tareas y el archivo de configuración config.js. En su interior tenemos las variables:

path
contiene las rutas a los archivos
environment
Coje su valor del entorno pasado por terminal o en su ausencia development
lintcss
Activa o desactiva el linter en la tarea css
server
La url del servidor local
targets
Entornos para los el módulo processhtml
options
opciones de configuración para los módulos instalados. Autoprefixer, imagemin, cssnano, processhtml (en modo desarrollo y en modo producción)

Las tareas están separadas por archivos gulp/tasks. Podemos llamarlas de forma individual:


gulp nombre_tarea

O utilizar npm para ejecutar cualquiera de las tres tareas principales.


npm run gulp-development

Ejecuta las tareas en modo desarrollo:

html
Realiza includes de los targets :dev (development).
css
Crear sourcemaps para facilitar el desarrollo, procesa sass en modo "expanded" para facilitar la lectura del código css, añade prefijos y notofica de los errores en el archivo. Revisa el css en busca de errores si lintcss esta activo
js
Revisa el código javascript en busca de errores.
jsMove
Desplaza el contenido de la carpeta vendor a dist.
img
traslada las imagenes pero no las comprime.
work
Abre la url del servidor y observa cambios en html,css,js y imagenes para ejecutar las tareas correspondientes.

npm run gulp-production

Ejecuta las tareas en modo producción:

cleanAll
Elimina todos los archivos de la carpeta /dist
html
Realiza includes de los targets :dist (production) y elimina los espacios en blanco del html.
css
Processa sass, añade prefijos css, limpia código, comprime css y renombra el archivo resultante con el sufijo .min
js
Revisa el código, concatena los archivos javscript, renombra el archivo resultante como script.min.js, lo comprime y notifica la finalización.
jsMove
Desplaza el contenido de la carpeta vendor a dist.
img
Comprime las imágenes y las traslada a la carpeta dist.

npm run watch

Ejecuta la tarea work para observar si tenemos cambios en el código.

Comandos npm

No es una guía sobre npm, pero dejo algunos comandos útiles para trabajar con la plantilla.

Comprueba si existen versiones más actuales de los modulos instalados:


npm outdate

Actualiza a los módulos a sus versiones más recientes:


npm update

Actualiza el modulo nombrado:


npm update nombre_modulo

Deinstala el modulo nombrado:


npm uninstall nombre_modulo

Una alternativa para tener los módulos actualizados es instalar globalmente el módulo npm-check-updates.


npm install -g npm-check-updates

Una vez instalado ejecutando ncu se actualizan todas las dependencias de nuestro proyecto y nos informa de los cambios en el terminal.


ncu

Para actualizar las dependencias y anotar los cambios en el package.json.


ncu -u

Existen más opciones para utilizar el módulo.

NOTA: Por algún motivo que no entiendo el comando ncu en windows funciona bien en powerShell pero no con gitbash. En este último el comando adecuado es ncu -u --packageFile package.json

tools
Dentro de la carpeta tools encontramos tres carpetas:
  • mixins
    • Cada mixin tiene su propio archivo. Los colocamos dentro de la carpeta mixin y los importamos en caso de necesitarlos en el archivo _tools.mixins.scs
  • functions
    • Es el mismo caso que los mixins. Las funciones estan separadas por archivos y las importamos desde el 'partial' _tools.functions.scss
  • animate
    • Animate es un ejemplo de donde ubicar una libreria de terceros. Esta libreria contiene mixins con diferentes animaciones en css. Cada animación tiene su carpeta y un archivo _tools.animate.scss donde hacemos la carga de aquellas animaciones necesarias para nuestro proyecto

En el interior de la carpeta tools. Junto a las carpetas mencionadas estan los archivos:

_tools.animate.scss
Archivo donde importamos las animaciones que utilizemos
_tools.functions.scss
Archivo donde importamos las funciones necesarias
_tools.helpers.scss
Archivo que contien placeholders comunes que podemos usar en múltiples proyectos
_tools.mediatool.scss
Archivo para desarrollo que asigna un background diferente para cada media query.
_tools.mixins.scss
Archivo donde importamos los mixins necesarios
Base
En la carpeta Base encontramos dos archivos:
  • _base.normalitze.scss : Archivo para unificar la visualización inicial de los navegadores.
  • _base.scss : En él escribimos los estilos css de los elementos html. Sin selectores de clase ni Id.
Layout
En layout escribimos únicamente la estructura general de la web. Si tiene uno o dos sidebars, el footer .... Volvemos a usar unicamente selectores de clase.
Components
Components es el apartado donde trabajaremos la mayoria del tiempo. En él van los estilos específicos de cada elemento. Un slider, un menu, un artículo... Si creamos un slider o un menú, les asignamos siempre archivos independientes para facilitar la reutilización y la localización.

Dentro de cada carpeta podemos colocar tantos archivos scss como sean necesarios. La nomenclatura de esos archivos debe empezar siempre con el prefijo de su carpeta. Es decir, dentro de components encontrariamos component.buttons.scss o dentro de layout layout.home.scss .

De este modo resulta más rápido localizar de donde procede cada selector.

Sintaxy y Formato

Stylelint se encarga de revisar el código css y su formato. En mi caso utilizo un plugin de sublime text para detectar los errores al momento. Si no queremos utilizar el linter en gulp basta con poner el valor (false) en la variable lintcss (config.js). Otro de los plugins interesantes es html css js Prettify. Se ocupa de formatear nuestro código. En su opciones de configuración podemos asignar los valores a seguir en multiples lenguajes. Como guía orientativa dejo unas cuantas opciones.

  • Identación de 4 espacios.
  • Cada selector y cada declaración tiene su propia línea.
  • Entre el selector y la llave de apertura poner un espacio.
  • Entre la propiedad y el valor un espacio despues de ":".
  • Las llaves de cierre tiene su propia línea.
  • Cada regla css tiene un linea en blanco de separación.
  • Siempre que se pueda, evitar el uso del selector de ID y selectores muy específicos como el adyacente.
  • Dentro de lo posible no escribir líneas con más de 80 caracters. Para facilitar esto sublime (view/ruler/80) nos muestra una línea con esa longitud. No lo utilizo de manera estricta pero si aproximada. De este modo facilitamos la lectura sin realizar scroll horizontal.

/*Regla css completa*/
.hola { /*selector*/
    position: absolute; /*propiedad/valor*/
    padding-bottom: 10px; /*propiedad + valor = declaración*/
} /*llave de cierre*/

    

/* MAL */
.selector{ /*Falta un espacio blanco entre selector y llave*/
position: relative; /*La declaración no tiene 4 espacios de identación*/
} /*Falta línea en blanco entre reglas*/
.selectordos {
position:relative; /*Falta espacio en blanco entre propiedad y valor*/
}

selectortres{ position:relative; /*La declaración y el selector no tienen su propia línea*/
display: block; background-color: green; /*La declaración no tiene su propia línea*/
}

.selectoruno, .selectordos, .selectortres { /*Cada selector no tiene su propia línea*/
    color: red;
} 

    

/* Bien */
.selector { /*Espacio blanco entre selector y llave*/
    position: relative; /*La declaración tiene 4 espacios de identación*/
} /*La llave de cierre tiene su propia línea*/
/*Línea en blanco de separación entre reglas*/
.selector-dos {
    position: relative; /*Espacio entre propiedad y valor*/
} /*LLave de cierre en su propia línea*/

selector-tres { 
    position: relative; /*La declaración tiene su propia línea*/
}

.selector-uno, 
.selector-dos, 
.selector-tres { /*Cada selector tiene su propia línea*/
    color: red;
}

Nomenclatura

La idea es seguir la nomenclatura Bem. Pero no es una regla. Usarla cuando sea de utilidad.


.nav,
.nav-menu,
.nav-general {
   /*Selectores que representan un bloque*/
}

.nav__item,
.nav-menu__item,
.nav-general__item {
    /*Selectores que representan elementos descendientes de bloques*/
}

.nav__item--active,
.nav-menu--open,
.nav-menu__item--open{
    /*Selectores que representan una alteración del estado inicial 
    de un bloque o de un elemento*/
}

Si quiero asignar un nombre compuesto a un selector utilizo el separador "-".

Preprocesadores

En caso de trabajar con preprocesadores ya sea sass, less o culaquier otro:

  • No superar tres niveles de identación.
  • Las variables globales siempre en _style-var.scss.
  • Utilizar nomenclatura descriptiva de la variable y no usar palabras propias de css o html.
  • Los include siempre la ultima o la primera declaración de la regla. De este modo resaltan entre las propiedades.

/*Ejemplo de errores*/
.menu {
    display: block;
    @include border-radius(20px); /*Mal*/
    margin: 10%;

    .menu__item {
        margin-left: 5%;

        .menu__item__span {
            color: $margin; /*Conduce a error. No usar palabras propias
             de css*/

            .menu__item__link { /*Exceso de anidación*/
                color: $pepito; /*Pepito no tiene ninguna relacion con 
                el color*/
            }
        }
    }
}
    
       
    

/*Buen uso*/
.menu {
    margin: 2em;
    @include border-radius(20px); /*include al final*/

    .menu__item {
        margin-left: 2em;

        .menu__item__span {
            color: $color_secundario; /*/                  
        }

    }

}
/*Lo sacamos de la anidación (máximo 3) para hacer más legible el código*/
.menu__item__link { 
    color: $color-link; /*la variable ya tiene un nombre que representa
     su función*/
}

HTML

Guía de estilos para html.

Sintaxy

No son normas inamovibles, son una referencia para unificar criterios entre todos aquellos que trabajen en el mismo proyecto.

Cada etiqueta que representa un bloque tiene su propio línea. De esta manera se facilita la lectura y la visibilidad de las etiquetas. Del mismo modo, cualquier etiqueta que contenga más de 80 caracteres en su interior le asignamos una línea propia. La idea es hacer más legible el contenido y ver fácilmente donde están las etiquetas de apertura y cierre.


<p>
El párrafo tiene las "p" con líneas independientes.
</p>
<ul>
  <li>Los elementos de la lista en la misma línea</li>
</ul>
  <!-- La"ul" que representa el bloque tiene su propia línea -->

Hacer uso de los espacios para diferenciar entre contenido.


<article>
 <p>Contenido de un artículo</p>
</article>
    <!-- Separación para recalcar cambio de tema -->
<article>
 <p>Contenido de otro artículo</p>
</article>

En html5 no es obligatorio el cierre de algunas etiquetas y no esta permitido en otras. Ninguno de los void elements (elementos vacíos) puede llevar etiqueta de cierre.


<img src="bla bla" alt=""/> <!--No es válido-->
<img src="bla bla" alt=""> <!--Correcto-->

Todas las etiquetas con cierre opcional se cierran.

     
<p>se cierra</p>
<ul>
  <li>se cierra</li>  
</ul>

Del mismo modo, html5 también permite omitir las etiquetas html, head y body bajo ciertas condiciones. Mientras sea opcional, las escribiremos siempre.

 
  
<!DOCTYPE html>
<html>
    <head>
      <meta charset="UTF-8">
      <title> </title>
    </head>
  <body>
    
  </body>
</html>

Usar comillas dobles para los atributos html.

Todas las imágenes deben llevar el atributo alt.

Todos los tags y sus atributos se escriben con minúsculas.


<img src="image/foto.png" alt="foto autor">
<img src='image/foto.png' alt="foto autor"> <!--Mal comillas simples-->
<img src="image/foto.png"> <!--Mal falta alt-->
<img SRC="image/foto.png"> <!--Mal atributo en mayúsculas-->

Las etiquetas <hr> y <br> se evitan siempre que sea posible.

Usar las mínimas etiquetas posibles en cada momento. A menos etiquetas menos trabajo tiene el navegador para renderizar.

Dentro de lo posible no superar los 80 caracteres por línea.

Siempre buscar la etiqueta con más significado semántico. Evitar la "divitis".


<figure>
  <img src="image/foto.jpg" alt="Montañas nevadas en la sierra">
  <figcaption>Paisaje de la sierra</figcaption>
</figure>

La identación en html es de 2 espacios. Utilizo menos identación que en css ya que suelo anidar más.

El atributo style en las etiquetas html no esta permitido.

La etiqueta input siempre debe llevar el atributo for y su label asociado. Recordar que la asociación se establece escribiendo el mismo valor en el for del label que el id del input.

 
<label for="name">Nombre</label> 
<input type="text" id="name">

Comentarios

Mientras trabajemos con gulp y el módulo processhtml utilizamos el sistema de comentarios del propio módulo. De este modo podemos comentar con exactitud en desarrollo y los eliminamos en producción.

Comentarios con processhmtl

Processhtml es un módulo para gulp con el que, entre otras cosas, realizar includes sin necesidad de plantillas.


<!-- build:css style.min.css -->
<link rel="stylesheet" href="css/style.css">
Sustituye el nombre de la hoja de estilos
<!-- /build -->

<!-- build:remove -->
Elimina cualquier contenido dentro del comentario.
<!-- /build -->

<!-- build:js:dist js/scripts.min.js -->
Solo en modo producción sustituye el src del script
<!-- /build -->

<!-- build:include include/footer.html -->
Inserta el archivo footer.html
<!-- /build -->

<!-- build:include:dist include/footer-dist.html -->
Solo en modo producción incluye el footer-dist.html
<!-- /build -->
